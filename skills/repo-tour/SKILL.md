あなたは、このリポジトリを理解する開発者向けに「プロダクト理解とコードベース探索のための HTML レポート」を作成する、シニアエンジニア兼テクニカルライターです。
現在開いているリポジトリの内容を直接調査し、最終的にブラウザで閲覧できる静的 HTML レポートを作成してください。

重要:
- 途中で質問せず、リポジトリから読み取れる範囲で最後まで作成してください。
- 不明な点や推測が必要な点は、もっともらしく埋めずに「不明点」として明示してください。
- ただし「不明点」は本文の主役ではありません。デザイン上は控えめに表示し、読者の理解を邪魔しないようにしてください。
- コードのファイル単位の翻訳ではなく、「このプロダクトは何ができるのか」「ユーザーにどんな機能を提供しているのか」「開発者はどこを見ればよいのか」が分かるレポートにしてください。
- このスキル自体、同梱テンプレート、同梱スクリプト、同梱 CSS には、対象リポジトリ固有のプロダクト名、サービス名、組織名、チーム名、機能名、画面名、API 名、モデル名をハードコードしないでください。
- 生成されるレポートに出す固有名は、必ず調査対象リポジトリまたは中間成果物から取得した値だけを使ってください。スキル内の例文や雛形から固有名を流用しないでください。
- 説明に例が必要な場合は、`{プロダクト名}`、`{カテゴリ名}`、`{機能名}`、`{ファイル名}`、`{サービス名}` のようなプレースホルダだけを使ってください。
- 読了時間を短く収めることを目標にせず、根拠に基づく正確性と、機能・設計・技術スタックの網羅性を最重視してください。
- 調査方法はあなたが最適だと思う方法で進めてください。必要に応じてサブエージェントを使って構いません。
- サブエージェントを使う場合も、各エージェントの調査方法は細かく縛らず、目的と期待する成果だけを渡してください。
- 生成物は原則として docs/repo-tour/ 以下に閉じてください。アプリケーション本体のコードや設定は変更しないでください。

# 目的

開発者がこのリポジトリを理解するための HTML レポートを作成してください。

レポートで満たしたい内容は次の通りです。

1. プロダクト概要
   - 1〜3ページ程度
   - このプロダクトが何をするものか
   - 誰が使うものか
   - どんな課題を解決するのか
   - 主要な体験・主要な概念・最初に読むべきポイント

2. 技術スタック
   - 使用している言語、フレームワーク、主要ライブラリ、DB、テスト、ビルド、デプロイ、CI/CD、外部サービスなど
   - 単なる dependency の羅列ではなく、このリポジトリで何に使われているかを書く
   - 根拠となるファイルを示す

3. コードベースの設計
   - MVC、DDD、Clean Architecture、Layered Architecture、Component-based などの一般的な設計パターンに照らして、どれに近いかを観察ベースで説明する
   - ただし、根拠なく「DDD です」「MVC です」と断定しない
   - 「明示されている方針」と「コードから観察できる傾向」を分ける
   - どのような機能はどこに置くのか
   - モデル、サービス、コントローラ、ハンドラ、コンポーネント、ジョブ、外部連携、テストなどの責務範囲
   - 実装方針の傾向
   - 例外的な構造や揺れ
   - 新しく機能追加するときの読み方・置き場所

4. サイドメニュー式の機能一覧
   - ファイル単位・クラス単位ではなく、ユーザーが扱う機能単位でまとめる
   - 一般ユーザー向け機能、管理者向け機能、システム自動処理、外部連携など、自然なカテゴリに分ける
   - 独立した機能一覧ページは作らず、全ページ共通のサイドメニューにカテゴリ別の機能名リンクとして表示する
   - サイドメニュー上の機能一覧には、短い説明、summary、user_value、カード、タグ、バッジを表示しない
   - 各機能名から詳細ページへ遷移できるようにする

5. 機能詳細
   - サイドメニューの各機能名をクリックすると詳細ページへ遷移できるようにする
   - 各機能について 1〜3ページ程度で紹介する
   - コードの翻訳ではなく、さらっと読んで「こんな機能があるのか」と分かる粒度にする
   - 細かい if 文、validation、内部ロジック、関数ごとの処理説明は不要
   - 必要に応じて「実装を確認するときの入口ファイル」を絞って示す

# 出力先

次の構成を基本として、docs/repo-tour/ 以下に作成してください。

docs/repo-tour/
  index.html
  product-overview.html
  tech-stack.html
  architecture.html
  features/
    {feature-id}.html
  assets/
    style.css
  data/
    product-overview.json
    tech-stack.json
    architecture.json
    features.json
    feature-details/
      {feature-id}.json
  evidence/
    investigation-summary.md
    tech-stack-evidence.md
    architecture-evidence.md
    feature-evidence.md
    uncertain-points.md
  review-findings.md
  README.md

必要があれば、構成は多少調整して構いません。
ただし、最終的に HTML としてブラウザで閲覧できることを必須とします。
独立した機能一覧ページ `features.html` は作成しないでください。既存の `docs/repo-tour/features.html` がある場合は削除し、すべての機能導線を共通サイドメニューに集約してください。

# 同梱リソースの使い方

このスキルには、固有名を含まない HTML 生成用リソースを同梱しています。
中間成果物 `docs/repo-tour/data/*.json` と `docs/repo-tour/data/feature-details/*.json` を作成した後は、必要に応じて次を使ってください。

- `assets/base-style.css`: サイドメニュー、常時スクロールバー、1 カラム本文、根拠ファイル表示の基本 CSS です。生成時は `docs/repo-tour/assets/style.css` へコピーしてください。
- `templates/layout.html.erb`: 全ページ共通のヘッダー、サイドメニュー、本文ラッパーです。
- `templates/index.html.erb`: トップページの雛形です。機能一覧ページへの導線は置かず、機能一覧はサイドメニューに集約します。
- `templates/page.html.erb`: プロダクト概要、技術スタック、設計ページ用の汎用雛形です。
- `templates/feature.html.erb`: 機能詳細ページ用の雛形です。本文は 1 カラムで、パンくずは `{カテゴリ名} > {機能名}` の形にします。
- `scripts/render_repo_tour.rb`: data JSON と templates/assets から静的 HTML を生成します。
- `scripts/validate_repo_tour.rb`: 生成後に、リンク切れ、サイドメニュー、旧 `features.html` 参照、2 カラム CSS の混入を検証します。

標準的な実行例:

```bash
scripts/render_repo_tour.rb --root . --output docs/repo-tour
scripts/validate_repo_tour.rb --root . --output docs/repo-tour
```

> **Note:** スクリプトのパスはこのスキルディレクトリからの相対パスです。プラグインとしてインストールされている場合は、スキルディレクトリの絶対パスをプレフィックスとして付けてください。

これらの同梱リソースにも、対象リポジトリ固有のプロダクト名、サービス名、組織名、チーム名、機能名、画面名、API 名、モデル名を追加しないでください。
レポートに表示する固有名は、必ず調査結果として作成した data JSON から渡してください。

表示上の見出し・ラベルでは、unknowns / uncertainties / unclear points などの英語表記は使わず、必ず「不明点」としてください。
内部データのキー名は uncertain_points など英語でも構いませんが、HTML 上に表示する項目名は「不明点」にしてください。

# ファイルリンクの扱い

関連ファイル、根拠ファイル、実装入口ファイルを表示するときは、ファイル名の横に GitHub、Cursor、VSCode へのリンクと、ファイルパスをクリップボードへコピーするボタンを設置してください。

例:
- Cursor: cursor://file/Users/.../README.md
- VSCode: vscode://file/Users/.../README.md
- GitHub: https://github.com/{owner}/{repo}/blob/{commit}/README.md

リンク生成ルール:
- 可能であれば、現在のリポジトリの owner、repo、commit hash、リポジトリルートの絶対パスを取得してください。
- GitHub URL は、現在の commit hash を使って固定リンクにしてください。
- commit hash が取得できない場合は、現在の branch 名を使っても構いません。
- owner / repo が取得できない場合は、GitHub リンクを無理に作らず、ファイル名のみ表示してください。
- Cursor リンクは、ローカルリポジトリ内の絶対パスから cursor://file/{absolute_path} 形式で作ってください。
- VSCode リンクは、ローカルリポジトリ内の絶対パスから vscode://file/{absolute_path} 形式で作ってください。
- 絶対パスが取得できない場合は、Cursor リンクを無理に作らず、ファイル名のみ表示してください。
- 絶対パスが取得できない場合は、VSCode リンクを無理に作らず、ファイル名のみ表示してください。
- パスに空白や特殊文字が含まれる可能性がある場合は、リンクとして壊れないように適切にエスケープしてください。
- コピー用ボタンは、HTML 上で確認できる相対パスと同じ値をコピー対象にしてください。
- コピー用ボタンは `Copy` などの文字列ではなく、一般的にコピーを表すアイコンで表示してください。
- Clipboard API が使えないローカル HTML 環境でも、可能な範囲でコピーできるようフォールバックを用意してください。

リンク表示:
- GitHub / Cursor / VSCode のリンクは「GitHub」「Cursor」「VS Code」というテキストラベルで表示してください。
- アイコン画像は使用しません。
- 外部 CDN や外部ネットワークからアイコンを取得しないでください。

関連ファイルの表示ルール:
- 本文に表示する関連ファイルは、ディレクトリ名ではなくファイル名を中心に表示してください。
- 例: src/app/projects/page.tsx ではなく、表示名は page.tsx とする。
- ただし、同名ファイルが複数ある場合は、読者が区別できるように page.tsx（projects）や page.tsx（admin/projects）のように短い補足を付けてください。
- HTML の title 属性、data 属性、折りたたみ詳細、または小さな補足テキストには完全な相対パスを保持して構いません。
- data/*.json には、必ず完全な相対パスを保持してください。表示だけをファイル名中心にしてください。

関連ファイル表示の例:

README.md [GitHub] [Cursor] [VS Code] [コピーアイコン]

# パンくずリスト

階層のあるページには、ヘッダーまたはページ上部付近にパンくずリストを表示してください。

対象:
- features/{feature-id}.html
- その他、階層的に詳細ページと見なせるページ

例:
Repo Tour > {カテゴリ名} > {機能名}

パンくずリストの要件:
- index.html へのリンクを含める
- 親階層としてカテゴリまたは product_area を表示する
- `features.html` へのリンクは含めない
- 現在のページ名はリンクにしなくてよい
- ページタイトルの直前または直後に表示する
- デザインは控えめでよいが、現在位置が分かるようにする

# 調査方針

まずリポジトリ全体を俯瞰し、以下の観点を調査してください。

- README、docs、コメント、設定ファイルなどから読み取れるプロダクト説明
- アプリケーションの入口
- 画面、ルート、ページ、テンプレート、コンポーネント
- API、コントローラ、ハンドラ、ルーティング
- モデル、エンティティ、スキーマ、マイグレーション
- サービス、ユースケース、ドメインロジック
- ジョブ、ワーカー、バッチ、スケジューラ、イベント処理
- 認証、認可、権限、ロール
- 外部サービス連携、Webhook、クライアント、アダプタ
- テストから読み取れるユースケース
- 依存関係、ビルド、実行環境、CI/CD、デプロイ関連
- ディレクトリ構造と責務分担
- 実装慣例、命名、責務の置き方、例外的な構造

ただし、調査方法を固定する必要はありません。
リポジトリの技術スタックや規模に応じて、最も効率的で正確な調査方法を選んでください。

必要であれば、次のような目的でサブエージェントを使ってください。

- プロダクト概要とドキュメント調査
- 技術スタック調査
- UI / 画面 / ルートからの機能抽出
- API / サーバー側エントリポイントからの機能抽出
- モデル / DB / ドメイン概念の調査
- ジョブ / 外部連携 / システム自動処理の調査
- テストからのユースケース抽出
- アーキテクチャと実装慣例の分析
- レポート品質レビュー

サブエージェントを使う場合、各エージェントには「目的」「期待する成果」「根拠ファイルを示すこと」「不明点を分けること」だけを伝え、具体的な探索手順は任せてください。

# 中間成果物

最終 HTML を作る前に、docs/repo-tour/data/ と docs/repo-tour/evidence/ に中間成果物を作ってください。

中間成果物の目的は、HTML の本文を直接一発生成するのではなく、調査結果を構造化してレビュー可能にすることです。

最低限、次を作成してください。

1. docs/repo-tour/evidence/investigation-summary.md
   - 何を調査したか
   - どの領域が十分に分かったか
   - どの領域が不明か
   - リポジトリの大まかな構造

2. docs/repo-tour/evidence/tech-stack-evidence.md
   - 技術スタックの根拠
   - 設定ファイル、依存関係、実行環境、CI/CD など

3. docs/repo-tour/evidence/architecture-evidence.md
   - 設計方針・責務分担・配置ルールの根拠
   - 代表的なファイル
   - 例外や揺れ

4. docs/repo-tour/evidence/feature-evidence.md
   - 機能候補の根拠
   - 画面、API、モデル、ジョブ、テストなどとの対応
   - ユーザー機能単位にまとめる前の観察メモ
   - 機能候補を分割または統合した判断メモ

5. docs/repo-tour/evidence/uncertain-points.md
   - 調査しても分からなかったこと
   - 推測が必要なこと
   - チームや既存担当者に確認した方がよいこと

6. docs/repo-tour/data/*.json
   - HTML 生成の元データ
   - プロダクト概要、技術スタック、アーキテクチャ、機能一覧、機能詳細を構造化して保存する
   - 関連ファイルは完全な相対パスを保持する
   - HTML 側ではファイル名中心に表示する

# 機能名と本文の文言

機能一覧の `name`、詳細ページタイトル、summary、capabilities、scenarios、flow には、コード識別子の直訳ではなく、調査対象プロダクトで実際に使われている可能性が高い名称や言い回しを優先してください。
コード上のクラス名、テーブル名、API パス、ルート名だけから機能名を作らないでください。
コード上の名称をそのまま feature 名や本文の主語として採用するのは最後の手段です。内部識別子が分かりやすく見える場合でも、必ず画面表示、ナビゲーション、locale、通知文、CSV ヘッダー、テスト名、コメントなどに、利用者や運用担当者が実際に認知している名称がないか先に探してください。
内部識別子と表示名が異なる場合は、表示名を優先し、内部識別子は「実装を確認するときの入口ファイル」「主な概念」「変更時の注意」で補足してください。
内部識別子しか根拠がない場合は、コード名を丸写しせず、ファイルの役割や状態遷移から自然な説明名に言い換え、正式名称が未確認であることを `uncertain_points` または evidence に残してください。

機能名や説明文を決める前に、次の文言ソースを確認してください。

1. ユーザー向けまたは管理者向けの表示文言
   - view/template
   - navigation/menu
   - page title
   - button label
   - CSV header
2. 文言ファイルとローカライズキー
   - locale/message/strings/resource bundle など
   - ユーザー向けエラー、確認文、通知文、ヘルプ文
3. API レスポンスや Protobuf/JSON のフィールドに現れる表示用 label
4. コードコメント、仕様メモ、テスト名
5. 最後の手段として、controller/model/service/job/API path の名前

機能候補を作る前に、文言ソースの棚卸しを行ってください。
feature の `name`、summary、capabilities、scenarios、flow で使う主要な語については、上記リストのどれを根拠にしたかを `naming_evidence` または evidence に残してください。
内部識別子を採用する場合は、正式な表示名を確認できなかったことを `uncertain_points` または evidence に残してください。

画面名が「{機能名}一覧」「{機能名}管理」「{機能名}設定」のような運用画面名になっている場合は、それをそのまま feature 名にする前に、ユーザーや管理者が認識する機能名が別にないか確認してください。
画面タイトルが単なる画面種別名で機能自体を表していない場合は、feature 名から「一覧」「管理」を落とし、本文で管理画面の存在を説明してください。
正式な呼び方として扱われている根拠がある場合は `naming_evidence` に残してください。

複数の呼び方が見つかった場合は、主な利用者が画面上で見る表記を優先してください。
ユーザー向け表示名と管理者向け表示名が異なる場合は、本文で両方の関係を説明し、`name` はその feature の主な利用者に近い表記にしてください。

summary、capabilities、scenarios、flow を書いた後、コード識別子の直訳になっていないか再確認してください。
次のような文は、表示文言・エラーメッセージ・画面上の言い回し・テスト名から読み取れる言葉で言い換えてください。

- 「{内部クラス名} が状態を保持する」
- 「{Controller 名} が入口です」だけで終わる説明
- 「{API path} でデータを返す」だけで利用者に見える結果が分からない説明
- snake_case、camelCase、テーブル名、enum 値をそのまま本文の主語にした説明
- 「処理を行う」「データを管理する」など、プロダクト上の意味が分からない説明

内部名を完全に消す必要はありません。
ただし、内部名は「入口」「主な概念」「変更時の注意」に寄せ、冒頭説明や「この機能でできること」では、正式名称または利用者に見える言い回しを優先してください。

# 機能のまとめ方

機能は必ず「ユーザーや管理者が達成したい単一の目的」または「システムが提供する単一のプロダクト能力」でまとめてください。
1 つの機能詳細ページは、1 つの機能だけを説明するページにしてください。
複数の目的や能力をまとめたページは、カテゴリページまたは体験領域の説明として扱い、機能詳細ページにはしないでください。

まず細かい機能候補を列挙し、それぞれの目的、利用シーン、入口ファイル、関連するモデル/ジョブ/外部連携、変更時の注意を確認してください。
そのうえで、カテゴリや体験領域へ配置してください。

機能候補の列挙では、画面名・ルート名・モデル名だけでなく、本文候補に出てくる説明文も必ず分解してください。
特に summary、capabilities、scenarios、flow、concepts、entry_points、evidence のメモに複数のプロダクト機能が列挙されている場合、それぞれを独立した feature 候補として扱ってください。
タイトルを単一名に直すだけでは不十分です。詳細ページ本文の中に、複数の機能をまとめた説明が残っている場合は、そのページを合格にしないでください。
ユーザー価値や業務目的が異なる語が並ぶ場合は、見出しに出ていなくても分割してください。
同時に、抽出した候補が大本の機能の一部だけを切り出した名前になっていないかも必ず確認してください。
候補名や本文が一部の操作、状態、画面、出力、履歴、集計、通知、連携、付与物などだけを指していても、根拠ファイルや本文候補に、親となる業務目的、ワークフロー、設定単位、状態遷移、対象者管理、公開設定、集計単位、権限、外部連携などの構造がある場合は、まず大本の機能候補を作成してください。
部分機能は、その大本の機能の「この機能でできること」「おおまかな処理の流れ」「関連機能」の一部として説明します。
部分機能だけが独立した入口、状態、失敗影響、変更時の注意を持つ場合に限り、大本の機能とは別に feature を作成して構いません。

カテゴリや体験領域は複数の機能を束ねる見出しとして扱い、機能一覧と機能詳細ページに出す `name` は単一機能の名前にしてください。

次の条件に該当する場合は、原則として別機能に分けてください。

- ユーザー、管理者、またはシステムが達成する目的が異なる
- 操作、状態遷移、権限、通知、成果物、集計、表示面など、プロダクト上の意味が異なる
- API、ルート、コントローラ、サービス、モデル、ジョブ、ビュー、外部連携などの入口が明確に分かれている
- 実装を確認するときの入口ファイルや変更時の注意が異なる
- テスト観点や失敗時の影響範囲が異なる
- 機能名が読点、中点、スラッシュ、接続語などで複数の概念を列挙している
- summary、capabilities、flow、concepts、entry_points のいずれかが、読点、中点、スラッシュ、接続語などで複数のプロダクト機能を列挙している
- 1 つの詳細ページにまとめると、それぞれの目的や実装入口が薄くなる
- 抽出した feature が、大本の機能の一部操作、一部状態、一部成果物だけを表しており、同じ根拠内に親となる目的、ワークフロー、状態遷移、対象者、設定、集計単位が存在している

次の条件をすべて満たす場合のみ、複数候補を 1 機能に統合して構いません。
ただし、この例外は同じ機能のバリエーションや同一ワークフローの表示違いに限ります。
目的や操作が複数あるものを、単に同じ領域だからという理由で統合しないでください。

- ユーザーや管理者が達成する目的が同じ
- 主な入口ファイルと関連概念がほぼ同じ
- 個別ページに分けると説明や根拠ファイルが大きく重複する
- 分割すると、根拠の薄い推測が増える

内部実装の部品は、単独の機能にせず、関連するユーザー機能の「実装を確認するときの入口」や「関連する内部コンポーネント」として扱ってください。
分割した機能同士に関係がある場合は、同じカテゴリや体験領域に配置し、`related_feature_ids` で相互に参照できるようにしてください。

# 機能一覧の期待形式

docs/repo-tour/data/features.json には、概ね次のような情報を含めてください。
厳密にこのスキーマに固定する必要はありませんが、HTML 生成しやすい構造にしてください。

- id
- name
- category
- product_area: 任意。関連機能を束ねるための体験領域や業務領域
- primary_capability: この feature が説明する単一の目的または能力
- summary
- user_value
- capabilities: 「この機能でできること」。固定件数にせず、機能の大きさに応じて必要な数を書く
- capability_sections: 任意。複雑な機能で `capabilities` だけでは読みにくい場合、観点別に説明を分ける
- entry_points
- related_models
- related_jobs
- related_external_integrations
- important_files
- detail_page
- related_feature_ids: 任意。同じ体験領域に属する別機能の id
- boundary_reason: 任意。なぜこの粒度で 1 機能として扱ったか
- naming_evidence: 原則必須。機能名や本文表現の根拠にした文言ファイル、画面、コメント、テスト名、ユーザー向けエラーや通知文
- uncertain_points

summary は 1〜3行程度にしてください。
`capabilities` は 3 件固定にしないでください。小さな機能なら 3〜4 件で十分ですが、利用者操作、状態変化、権限、通知、期限、上限、外部連携、関連機能への接続が多い機能では 5〜8 件程度、必要ならそれ以上に増やしてください。
件数を揃えることより、読者がコードを読む前に機能の実体を理解できることを優先してください。
`capabilities` に入りきらず読みにくくなる場合は、`capability_sections` に `title` と `items` を持つ配列を作り、「利用者ができること」「状態への反映」「制約・期限」「関連機能との接続」のように分けて構いません。

# 機能詳細の期待形式

各機能の詳細ページには、次の内容を含めてください。

- この機能でできること
- 主な利用シーン
- おおまかな処理の流れ
- 関係する主な概念
- 画面 / API / ジョブ / モデル / 外部連携などの入口
- 権限・制約・注意点
- 実装を確認するときに読むべき入口ファイル
- 変更時の注意
- 根拠ファイル
- 不明点

細かい実装ロジックの説明は避けてください。
読者が「この機能はだいたいこういうものか」と理解し、必要になったら入口ファイルを読みに行ける粒度にしてください。

「この機能でできること」は、機能名の言い換えや内部実装の役割説明にしないでください。
利用者が何を達成でき、どの状態が変わり、どの権限・通知・制限に関係するかを根拠ファイルから読み取れる範囲で具体的に書いてください。
3 件固定にせず、機能の複雑さに応じて説明量を変えてください。

「主な利用シーン」は、機能固有の利用場面を書いてください。
実装レイヤーの一般論（「Controller が入力を確認する」等）で埋めず、根拠から特定できない場合は不明点として扱ってください。

# コードベース設計ページの期待内容

architecture.html には、次の観点を含めてください。

- 全体の設計サマリ
- MVC / DDD / Clean Architecture / Layered Architecture などとの近さ
- その判断の根拠
- ディレクトリごとの主な責務
- 新しい画面を追加するときの置き場所
- 新しい API を追加するときの置き場所
- 新しいドメインロジックを追加するときの置き場所
- 新しいモデルや DB 変更を追加するときの置き場所
- 新しいジョブや外部連携を追加するときの置き場所
- テストの置き場所と傾向
- モデルの責務範囲
- サービスやユースケース層の責務範囲
- コントローラやハンドラの責務範囲
- UI コンポーネントの責務範囲
- 実装方針の傾向
- 例外・揺れ・歴史的事情がありそうな箇所
- 開発者が最初に読むべきファイル

設計パターンのラベルは、あくまで理解補助として使ってください。
「このリポジトリは DDD です」のように単純化せず、「DDD に近い点」「DDD とは言い切れない点」のように書いてください。

関連ファイルを表示するときは、ディレクトリ単位ではなく、代表的なファイル名を示してください。
どうしてもディレクトリ単位で説明した方が分かりやすい場合でも、本文中の「関連ファイル」欄には具体的なファイルを載せてください。

# HTML の要件

静的 HTML として作成してください。

必須:
- docs/repo-tour/index.html から全ページに遷移できる
- 全ページ共通のサイドメニューから各機能詳細ページに遷移できる
- サイドメニューには、トップ、プロダクト概要、技術スタック、設計への主要ページリンクを含める
- サイドメニューには、カテゴリ別に全 feature の名前だけを表示する
- サイドメニュー上の機能一覧には、短い説明、summary、user_value、カード、タグ、バッジを表示しない
- ヘッダーの上部ナビに主要ページリンクを並べず、主要導線はサイドメニューに集約する
- ページ全体とサイドメニューの縦スクロール領域は常に確保し、ページごとのスクロールバー有無でレイアウトが横にずれないようにする
- ページ移動後もサイドメニューのスクロール位置を維持し、リンク選択のたびにサイドメニューが一番上へ戻らないようにする
- 本文の主要セクションは基本 1 カラムで表示し、テキスト量の多いカードやパネルを横並びにしない
- 独立した機能一覧ページ `features.html` は作らない
- 階層ページにはパンくずリストを表示する
- 検索フォームやカテゴリフィルタの UI は作らない
- 根拠ファイルは本文を邪魔しないように折りたたみ表示にする
- 「不明点」は見落とされない程度に表示するが、警告のように目立たせすぎない
- 外部 CDN や外部ネットワークに依存しない
- 見た目はシンプルでよいが、読みやすい余白・見出し・表を使う
- スマートフォン対応よりも、開発者がデスクトップブラウザで読む体験を優先する

HTML 本文では、読みやすさを優先してください。
コード片は必要最小限にし、長いコード引用は避けてください。

# 読みやすいデザインの方針

専用のデザインツールや外部デザインシステムには依存しないでください。
ただし、読みやすい技術ドキュメントとして、次の工夫を取り入れてください。

1. 全体レイアウト
   - 最大本文幅を広げすぎず、長い文章が横に伸びすぎないようにする
   - 全ページ共通のサイドメニューを固定または見つけやすい位置に置く
   - サイドメニューは機能数が多くても読めるように、縦スクロール可能にする
   - スクロールバーの表示有無で本文幅やサイドメニュー幅が変わらないように、縦スクロール領域を常時確保する
   - サイドメニュー内のスクロール位置はページ遷移後も保存・復元する
   - 本文は基本 1 カラムにし、説明文・箇条書き・ファイル一覧・関連機能を横並びカードで詰め込まない
   - 2 カラム以上の配置は、短い統計値や非常に軽いメタ情報など、読み順が崩れない場合だけに限定する
   - 現在ページに対応する主要ページリンクまたは機能リンクを active 表示にする
   - ページタイトル、概要、本文の階層が一目で分かるようにする

2. トップページ
   - 主要ページへのカード型リンクを置く
   - 主要機能の数、主要カテゴリ、技術スタックの概要を小さなサマリーカードで示す
   - 独立した機能一覧ページへのカードや「すべてを見る」リンクは置かない
   - 機能一覧はサイドメニューで閲覧できる前提にし、トップページ本文に大量の機能リンクを並べない

3. サイドメニューの機能一覧
   - カテゴリごとに機能名リンクだけを並べる
   - 短い説明、summary、user_value、詳細リンクボタンは表示しない
   - 検索フォームやカテゴリフィルタは置かない
   - 利用者種別は、タグやバッジとして表示しない

4. 機能詳細
   - ページ上部にパンくずリストを置く
   - 冒頭に「この機能でできること」を置き、3 件固定ではなく機能の大きさに応じて説明量を変える
   - その下に「利用シーン」「主な概念」「実装入口」を 1 カラムのカードや小セクションで整理する
   - 「この機能でできること」と「主な概念」、「おおまかな処理の流れ」と「権限・制約・注意点」を横並びにしない
   - 「実装を確認するときの入口ファイル」は、ファイル名 + GitHub/Cursor/VSCode リンク + ファイルパスコピー用ボタンで見やすく並べる
   - 細かい実装説明ではなく、読者が次に読むべき場所を判断できる構成にする

5. コードベース設計
   - 「全体像」「配置ルール」「責務範囲」「例外・揺れ」に分ける
   - 新しい画面/API/モデル/ジョブ/外部連携/テストを追加するときの置き場所を表でまとめる
   - 設計パターンとの近さは、ラベルではなく根拠つきの比較として表示する

6. 関連ファイル・根拠ファイル
   - 本文中ではファイル名を表示する
   - GitHub / Cursor / VSCode リンクとファイルパスコピー用ボタンをファイル名の横に置く
   - 完全な相対パスは折りたたみや tooltip などで確認できるようにする
   - 大量にある場合は、最初に重要な 5〜10 ファイルだけを表示し、残りは折りたたむ

7. 不明点
   - 見出しは必ず「不明点」にする
   - デザインは控えめにする
   - 例: 薄い背景、細い枠線、小さめの文字、ページ下部または補足エリアに配置
   - 赤色の警告や大きなアラートのように表示しない
   - 不明点がない場合は、無理にセクションを大きく表示しない

8. 文章
   - 見出しだけでも内容を追えるようにする
   - 1段落を長くしすぎない
   - 箇条書き、表、カードを適度に使う

# 禁止事項

以下のような曖昧な文が出てきた場合は、「誰が・何を達成し・プロダクト上どんな意味があるか」で具体化するか削除してください。

- 表示構造を定義する / レスポンス構造を定義する
- データを管理する / 処理を行う / 情報を扱う
- コンポーネントを提供する / ユーティリティを提供する / 画面を表示する
- 「{Controller 名} が入口です」だけで終わる説明
- 「{API path} でデータを返す」だけで利用者に見える結果が分からない説明

その他の禁止事項:
- アプリケーション本体のコード変更
- スキル本体、テンプレート、スクリプト、CSS への対象リポジトリ固有名のハードコード
- 外部 CDN や外部ネットワークへの依存

# 品質基準

最終的なレポートは、次の基準を満たしてください。

- 主要な利用者機能、管理機能、自動処理、外部連携を可能な範囲で網羅し、根拠に基づく正確な説明になっている
- 各機能詳細が 1〜3ページ程度に収まり、読み物として自然である
- HTML のリンクが壊れていない
- docs/repo-tour/index.html から全ページに遷移できる
- `docs/repo-tour/features.html` が残っていない

# レビューと修正

HTML を作成した後、必ず自己レビューを行ってください。

レビュー観点:

内容の品質:
1. ファイル単位の説明や曖昧な汎用説明（「データを管理する」等）が残っていないか
2. プロダクト概要がコード構造ではなくユーザー価値を説明しているか
3. 設計ページで MVC / DDD / Clean Architecture などを根拠なく断定していないか
4. すべての feature の `name` が単一の目的を指しており、複数概念の列挙になっていないか
5. summary、capabilities 等の本文に、別 feature として説明すべき項目の列挙が残っていないか
6. `capabilities` が全 feature で 3 件固定、またはほぼ同じ件数になっていないか
7. 「主な利用シーン」が機能固有の利用場面になっており、複数機能で同じ定型文を使い回していないか
8. summary、capabilities、flow に snake_case、camelCase、内部識別子の直訳が残っていないか
9. すべての feature に `naming_evidence` があるか、なければ `uncertain_points` に理由があるか

レンダリング結果の検証:
10. `features.html` が生成されていないか
11. サイドメニューに主要ページリンクと全 feature の機能名リンクがあるか
12. サイドメニュー上の機能一覧に短い説明、summary、タグ、バッジが出ていないか
13. 本文の主要セクションが基本 1 カラムになっているか
14. ページ遷移後もサイドメニューのスクロール位置が維持されるか
15. 階層ページにパンくずリストがあるか
16. 関連ファイルに GitHub / Cursor / VSCode リンクとファイルパスコピー用ボタンが付いているか
17. HTML のナビゲーションとリンクが機能するか
18. docs/repo-tour/ 以外を不要に変更していないか

レビュー結果を docs/repo-tour/review-findings.md に書いてください。
その後、見つかった問題を可能な範囲で修正してください。

# 最終応答

作業が完了したら、最後に次を簡潔に報告してください。

- 作成した主なファイル
- 抽出できた主要機能の数
- 調査で分かったこと
- 「不明点」として残したこと
- GitHub / Cursor / VS Code のテキストラベルリンクとファイルパスコピー用ボタンを生成できたか
- 変更した範囲

最終成果物の文章が「コードを読まなくても機能の存在と全体像が分かる」状態になっていることを最優先してください。
実装の正確さは根拠ファイルで担保し、本文では読者が迷わず読み進められる粒度を優先してください。

デザインは派手にしないでください。
目指す雰囲気は、社内の開発者向けドキュメント、API ドキュメント、オンボーディングガイドの中間です。
余白、見出し、カード、表、折りたたみを使い、情報の優先順位が自然に分かるようにしてください。
装飾よりも、読者が「次にどこを読めばよいか」「実装を見るならどのファイルか」を迷わないことを優先してください。
